# Multiplayer Server Architecture

## Ownership model

The authoritative service separates transport connections, resumable sessions, lobby presence, room membership, matchmaking tickets, and game sessions. A player session may own one active connection, one room, and one game. A transport loss preserves resumable identity for the configured grace period.

## Modules

| Area | Responsibility |
| --- | --- |
| `ConnectionManager` | Socket limits, payload bounds, backpressure, heartbeat, connection lookup |
| `SessionManager` | Identity, rotating reconnect credentials, resume state, idle cleanup |
| `LobbyService` | Presence, snapshots, room summaries, bounded lobby chat |
| `RoomManager`, `Room` | Membership, host transfer, settings, passwords, readiness, room chat |
| `MatchmakingQueue` | Unique tickets, matching, confirmation, timeout and cancellation |
| `GameSessionManager`, `GameSession`, `GameTimer` | Per-game engine, player views, event sequence, deadline and completion |
| `Store` interfaces | Serializable sessions, rooms, matches, profiles, in-memory implementation |
| `RoomOwnership` | Node ownership and future message-bus boundary |
| `RateLimiter` | Connection, chat, room creation, and command budgets |
| `MetricsService` | Aggregate operational counters without personal data |

## Isolation

Every room and game has a unique ID, seed, serializable state, event sequence, timer, and member set. Broadcast targets are derived from membership rather than a global socket list. Player views are generated by the shared engine and hide opponent hand, deck order, unrevealed cards, and private credentials.

The server has no singleton game, turn timer, or shared player array. Timer callbacks resolve through the owning game ID and are cancelled at lifecycle end. Duplicate request IDs are bounded and return the original result or a structured error without repeating a game action.

## Protocol and operations

All envelopes use `aeonfront/3`, a request ID, and scoped room or game IDs when required. Runtime validation rejects oversized, malformed, stale, or cross-scope messages. Fatal protocol and catalog mismatches stop the session before game entry.

`/health` reports liveness. `/ready` confirms catalog and service readiness. Optional `/metrics` reports aggregate connections, lobby users, rooms, games, queue size, message and error totals, duration, reconnects, and timeouts without names or chat content.

## Current deployment boundary

The released implementation uses an in-memory store and one process, but room ownership, serializable records, store interfaces, and message-bus boundary are explicit. External databases and multi-node routing are not required for the current 1v1 release and are not emulated inside game rules.
